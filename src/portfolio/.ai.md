# src/portfolio/

## 목적
복수 전략 출력을 **한 곳에서 조립** 해 포트폴리오 리스크 모듈 (`src/risk/`) 로 흘려보내고, 주문 의도를 정책과 최신 리스크 리포트에 대해 게이팅하는 레이어.

## 구조 (#78 완료 후, #94 확장)

2-class composition 구조 (D1):
- `orchestrator.py` — `_SyncStrategyOrchestrator` (private, lock-free sync, #70 stub rename). 하위 호환 alias: `StrategyOrchestrator = _SyncStrategyOrchestrator`. `compute_fear_greed_proxy(price_history, window=252)` 헬퍼 (#87 P4) — 가격 기반 `close_latest / rolling_max(window)` 정규화, 소셜 크롤 없음.
- `_async_orchestrator.py` — `AsyncStrategyOrchestrator` (public async, #78). `_SyncStrategyOrchestrator` 를 composition 으로 감쌈.
- `config_loader.py` — `load_orchestrator_from_yaml(path, policy) -> AsyncStrategyOrchestrator` (#94). YAML 파싱 → importlib 동적 클래스 로드 → `metalabeler.load_path` 치환 → 전략 등록. strategy_id 중복 시 ValueError. MetaLabeler.load 실패 시 RuntimeError (fail-fast).
- `_strategy_adapter.py` — `_StrategyAdapter` (#94). sync `on_bar(bar, history, context)` → 오케스트레이터 `on_bar(ctx)` 어댑터. `MomoBtcV2` 등 sync Protocol 전략을 AsyncStrategyOrchestrator 에 연결.
- `order_intent.py` — `OrderIntent` dataclass (frozen=True, slots=True). `run_bar` 반환 타입. **#238 Item 7**: `reduce_only: bool = False` 필드. orchestrator 가 strategy SELL 마다 `reduce_only=True` 설정 (불변식: 우리 전략은 전부 long-only → sell=롱 청산, 숏 진입 의도 절대 없음). `src/live/conversion.intent_to_order_request` 가 `OrderRequest.reduce_only` 로 전달 → Binance 가 롱 없으면 no-op (맨몸 숏 진입 불가). 근본 사고(롱전용 sell 이 naked short 둔갑) 거래소단 차단. default False=하위호환. **post-only Maker (2026-05-22, post-only-maker-entry)**: `entry_order_type: Literal["market","post_only"]="market"` + `ref_price: float|None=None` 필드. `run_bar` 가 BUY 진입에 한해 strategy 의 `entry_order_type` 속성을 stamp (SELL 은 항상 market), ref_price 에는 per-symbol `order_price` 를 stamp (멀티심볼 정확성). executor 가 post_only 면 GTX LIMIT 으로 발주. default("market", None)=레거시 byte-identical.
- `sizing.py` — `resolve_size(signal, recent_returns)`. Signal 메타데이터 → Kelly 분기. `src/risk/sizing.py` (Kelly 수식) 와 다른 개념. **#238**: `size_to_qty(fraction, *, equity, price, symbol) -> float | None` 추가 — `resolve_size` 가 돌려주는 값은 *가용자본 비율* [0,1] 인데 orchestrator 가 이전엔 그 비율을 코인 수량으로 **직접** 사용 (size=0.05→0.05코인, momo full size=1.0→1.0 BTC≈$80k → -2019 Margin-insufficient 폭주의 근본 원인). `qty = (fraction*equity)/price` 변환 후 거래소 필터 적용: LOT_SIZE step ROUND_DOWN (`src/live/conversion.get_step_size` 재사용, lazy import 로 portfolio↔live 순환 회피), MIN_NOTIONAL (Binance USDT pair 만 — 보수적 상수 `BINANCE_MIN_NOTIONAL_USDT=5`, exchangeInfo 폴링 안 함 — hot-path + 보장-거부 주문은 #238 가 잡은 버그류), zero-qty/미지원심볼/비정상입력 → `None` (드롭, OrderIntent 미발행). fraction>1 은 1.0 으로 clamp.
- `binance_universe.py` — Binance USDT spot **30종 단일 source of truth** (2026-05-21 fix). `BINANCE_USDT_TOP30: tuple[str,...]` + `PIN_DATE: str` + `get_universe() -> list[str]`. dashboard (`cs_tsmom_signals._refresh()`) + live (`scripts/live_run.py` 의 binance-testnet-shadow universe provider) + backtest 가 모두 이 한 모듈을 import. 갱신 시 여기 한 곳만 수정. **버그 히스토리**: 이전엔 dashboard 가 `bench_cs_tsmom_crypto.fetch_top_universe(30)` 으로 매 호출마다 24h 거래량 상위 30 을 동적으로 잡아 메이저(BTC/ETH/SOL)가 변동으로 빠지면 NaN score 양산 + BUY 후보 0. 두 곳이 분리되어 inconsistency 가 surface 안 되고 잠복했음. 6개월마다 (5/1, 11/1) 갱신 + spec `docs/specs/strategies/cs-tsmom-crypto-daily.md` 동시 갱신.
- `__init__.py` — 공개 API: `AsyncStrategyOrchestrator`, `OrderIntent`. `_SyncStrategyOrchestrator` 는 private.

## 역할
- 전략 N개의 일수익률 → T×N DataFrame → `risk.compute_portfolio_risk_from_df`
- 생성된 `PortfolioRiskReport` 를 `Snapshot.portfolio_risk` 로 주입
- 가격 히스토리 → `compute_fear_greed_proxy` → `Snapshot.fear_greed_proxy` 로 주입 (`evaluate_order()` 에서 조립, `refresh_portfolio_risk()` 시그니처 불변)
- `risk.evaluate(policy, snap)` 한 번 호출로 per-order 판정 + per_portfolio_risk 판정을 **같은 precedence chain** 에서 처리 (`extreme_fear_block` 평가 시 `snap.fear_greed_proxy` 직접 참조)

## 현재 스코프 (#70, #76 에서 제공)
- Sync API: `register_strategy_returns`, `refresh_portfolio_risk`, `evaluate_order`
- `strategy_reliability_score(strategy_id)` → [0,1]: `convex_base × drawdown_gate`. NaN 가드: T<20 → 0.0. KR101139626B1 differs: 곱셈 게이트(불연속 지시함수) ≠ KR 가산합.
- **reliability metric**: drawdown_gate = clip(1 − max_dd/0.20, 0, 1). max_dd ≥ 20% → 하드 제로.
- 2+ 전략이 있어야 report 생성. 1전략 이하 / T<2 → `ALLOW` 경로 유지 (회귀 zero)

## 완료 (#78)

### run_bar (async tick driver)
- `AsyncStrategyOrchestrator.run_bar(ts, market_snapshot)` — 등록된 모든 전략을 `asyncio.gather` 로 동시 실행, 각 전략 출력에 `resolve_size` + `risk.evaluate` 를 적용하고 ALLOW 된 것만 `list[OrderIntent]` 로 반환.
- 동기 전략(`on_bar` 는 일반 함수): `asyncio.to_thread` 로 스레드풀에서 실행.
- 비동기 전략(`on_bar` 는 코루틴): `asyncio.create_task` 로 직접 실행.
- 전략 구분은 `inspect.iscoroutinefunction(strategy.on_bar)` 로 dispatch.

### Fraction→qty 변환 배선 (#238)
- `run_bar` 의 주문 조립 지점에서 `fraction = resolve_size(...)` → `qty = size_to_qty(fraction, equity=venue_equity, price=order_price, symbol=order_symbol)`. `qty is None` 이면 `continue` (OrderIntent 미발행 + `portfolio.orchestrator.size_drop` 로그). 변환된 `qty` 가 risk `Order` + `OrderIntent` 양쪽에 사용 → `OrderIntent.qty` 가 실제 코인/주식 수량. `src/live/conversion.intent_to_order_request` 의 step ROUND_DOWN 은 최종 가드로 그대로 유지 (제거 안 함).
- **Venue-aware equity (불변식)**: `order_symbol` 이 `*USDT` (len>4) → `market_snapshot["equity_usdt"]`; 그 외 (KRX 6자리 등) → `market_snapshot["equity_krw"]`. 키 부재 시 0.0 → `size_to_qty` 가 드롭 (잘못된 통화로 주문하느니 무주문이 안전). **현재 ambiguity (사람 결정 필요)**: `_tick_to_market_snapshot` (`src/live/loop.py`) 와 `SnapshotBuilder` 는 `equity_krw` 를 placeholder(100000.0)로만 채우고 `equity_usdt` 는 채우지 않음. 실잔고는 `src/dashboard/account_info.py::AccountInfoProvider._fetch_binance` 의 `available_usdt` 에 있으나 dashboard 전용으로 snapshot 에 미배선. **권장 후속 배선**: `SnapshotBuilder` 가 `AccountInfoProvider` (또는 broker 잔고 조회) 를 주입받아 `build_snapshot` 에서 `equity_usdt`/`equity_krw` 를 실값으로 채움 — 본 PR 은 순수 함수 + orchestrator 배선 + snapshot 키 계약만 확정, 실잔고 source 배선은 별도 결정.

### 격리 / 쿼런틴 (circuit-breaker)
- 전략이 예외(RuntimeError, asyncio.TimeoutError 등) 를 3회 연속 발생시키면 `_quarantined` set 에 추가 → 이후 `run_bar` 에서 skip.
- 성공 시 해당 전략의 실패 카운터(`_fail_count`) 를 0으로 reset → 쿼런틴 조건 재시작.
- `quarantined_strategies` property 로 현재 쿼런틴 목록 조회 가능 (frozenset).

### 백그라운드 리프레시 (bar-clock / wall-clock)
- **bar-clock 모드**: `AsyncStrategyOrchestrator(policy, refresh_every_n_bars=N)` — `run_bar` 가 N번 호출될 때마다 `refresh_portfolio_risk_async` 자동 실행.
- **wall-clock 모드**: `start_risk_refresh_loop(interval_sec, jitter_frac)` / `stop_risk_refresh_loop()` — 백그라운드 asyncio Task 로 주기적 리프레시. 지터(±jitter_frac) 포함.
- `refresh_portfolio_risk_async` 는 `asyncio.Lock` 으로 동시 쓰기 보호.

### Live-scanner stop/TP 매니저 (#227 S2)

- `live_position_risk.py::LivePositionRiskManager` — 매 tick `evaluate(symbol, last_price, ts)` 호출 시 보유 (strategy_id, symbol) 페어를 `StrategyPositionStore` 에서 조회 + 평균진입가를 `PnLAggregator._cost_basis` 에서 조회 → stop/TP/trailing 임계값 breach 시 청산 `OrderIntent` emit + WAL `position_stop_triggered` 기록.
- 정책 등록: `register_strategy_policy(strategy_id, stop_loss_pct, take_profit_pct, trailing_stop_pct=None)`. live_run.py 의 `_register_exit_policies` 가 (1) LiveScannerMixin 전략 class attr (default 0.03/0.06) (2) `stop_loss_pct`+`take_profit_pct` 를 명시한 단일종목/universe 전략 (예: momo-btc-v2, production.yaml kwargs) 둘 다 wire-up. 둘 다 아니면 skip (임계값 발명 금지).
- **LONG/SHORT 양방향 (#238)**: `_lookup_position` 은 부호 있는 held 반환 (양수=long, 음수=short; 0 일 때만 flat). LONG = 기존 동작 bit-identical (stop=가격↓, tp=가격↑, trailing=high-water, 청산=`side="sell"`). SHORT = 전부 반전 (stop=진입가 +stop_loss_pct 위로 상승, tp=진입가 -take_profit_pct 아래로 하락, trailing=low-water 추적 후 +trailing_stop_pct 반등, 청산=`side="buy"` cover). 청산 qty 는 항상 `abs(held)`. 근본 사고: momo-btc-v2 가 naked -1 BTC short 를 ZERO auto-stop 으로 운용 (매니저가 long-only 였음).
- 청산 우선순위: `stop_loss` → `take_profit` → `trailing_stop` (LONG trailing 은 water_mark > entry, SHORT trailing 은 water_mark < entry 일 때만 작동, stop_loss 와 중복 방지).
- 모든 산술 `Decimal` — `PnLAggregator` 정합성. `PnLAggregator._apply_to_cost_basis` 도 #238 에서 short 진입가 추적 추가 (sell + held<=0 → 음수 측 평균; LONG 매도 경로 bit-identical). water_mark `dict[(sid, symbol), Decimal]` 는 보유 0 도달 시 자동 클린.

### Live-scanner per-symbol dispatch (#227 S1)

- `run_bar` 의 strategy dispatch 분기에서 `getattr(strategy, "is_live_scanner", False)` 가 True 이고 `market_snapshot["ohlcv_history"]: dict[symbol, DataFrame]` 가 비어있지 않으면 universe 의 각 symbol 마다 별 task 를 spawn — 각 task 는 단일 symbol 의 `market_snapshot = {symbol, history, price, equity_krw}` 와 `factors = universe_factors.get(symbol, {})` 를 받음.
- legacy 단일 dispatch path (cs_*, momo_*) 는 변경 없음 — `is_live_scanner` 미선언 + `ohlcv_history` 부재 시 동일 ctx 로 단일 호출.
- per-symbol 결과 OrderIntent 는 `task_symbols` parallel list 로 추적 → 적합한 symbol 로 발행. legacy 는 `task_symbols[i] is None` → `market_snapshot["symbol"]` fallback.
- **격리 (quarantine) dedup**: live-scanner 가 N 종목 모두에서 throw 시 `_fail_count` 가 1만 증가 (consecutive-bad-tick 의미 보존). reset 도 같은 tick 에 sibling task 가 throw 했으면 보류.
- **중복 주문 backstop (#238 Item 3)**: `min_order_interval_sec` (생성자 kwarg, **기본 0.0=비활성** → 기존 테스트·백테스트·universe-scan 리밸런스 bit-identical). >0 이고 비-live-scanner 전략이 동일 `(sid, symbol, side)` 를 wall-clock window 안에 재발사하면 `decision="hold" reason="duplicate_order_throttled"` 로 skip. 다른 action(reversal/exit)은 별 key → 절대 throttle 안 됨. live-scanner 는 `_live_entered` lifecycle 가 관장하므로 제외. Item 1(전략 레이어 throttle)의 orchestrator 레이어 방어선. **라이브 wiring(live_run.py orchestrator 생성자)은 Item 2 작업과 파일 충돌 회피 위해 후속.**

### Live-scanner 진입 상태 `_live_entered` (#238, 2026-05-22)
- `_live_entered: set[(sid, symbol)]` — live-scanner 가 진입한 (전략, 종목). dispatch 가 BUY 신호를 보고 이 set 에 있으면 `decision="hold" reason="live_position_open"` 으로 차단 (1 position per strategy-symbol).
- 추가: dispatch 진입 시. 제거: `release_live_position` (stop/TP 청산 — `cooldown_after_stop_sec>0` 이면 `_stop_cooldown_until` 도 기록). `restore_live_entered` 가 부팅 시 store position 으로 복원.
- `sync_live_entered(sid, symbol, qty)` (2026-05-22) — `PositionReconciler` 의 auto-fix 콜백. reconciler 가 broker↔store 정합 시 `_live_entered` 도 동기화 (qty=0→discard, qty≠0→add). **미동기화 시 reconciler 가 청산한 phantom 이 set 에 영구 박제 → 영구 진입 차단 회귀** (restore_live_entered ↔ reconciler 비동기화 = 재시작 후 11h 매수 0 사고). `_stop_cooldown_until` 은 안 건드림 — reconcile sync 는 stop 청산이 아니므로 churn cooldown(#266) 우회 금지.

### Runtime ON/OFF 토글 (#180)
- `enable_strategy(id)` / `disable_strategy(id, *, positions=None)` — runtime 활성/비활성 전환. `_disabled: set[str]` 관리.
- `is_enabled(id) -> bool`, `disabled_strategies` (frozenset) property.
- `run_bar` 가 `_disabled` 항목을 skip — 신규 시그널 차단.
- **즉시 청산 (D1)**: `disable_strategy(id, positions=[(symbol, qty), ...])` 가 각 포지션에 대해 `OrderIntent(side='sell', reason='strategy_disabled_liquidation')` list 반환. caller 가 broker 로 발주.
- **WAL 감사 로그**: 생성자 `wal_observer: Callable[[WALEvent], None] | None` 받음. 활성 전이 시 `WALEvent(event_type='strategy_toggled', payload={strategy_id, enabled, actor='user'})` 호출. observer 예외는 swallow + log warn.
- **Idempotent**: 이미 disable 인 전략을 재차 disable → state 유지, WAL 재발행 X. positions 가 주어지면 청산 intents 는 매번 새로 생성 (caller 가 최신 보유분을 주입한다고 가정).

### Strategy 평가 가시화 — `strategy_evaluated` WAL event (#231 S5)
- `run_bar` 의 매 strategy dispatch 결과마다 `EVENT_STRATEGY_EVALUATED` (`src/live/types.py`) WAL event 1건 emit. live-scanner per-symbol dispatch 의 경우 `(strategy_id, symbol)` 쌍마다 1건 — 30종목 universe + 5 strategy = tick 당 150 events.
- payload: `{strategy_id, symbol, decision, reason}`. decision 값:
  - `"buy"` / `"sell"` — `Signal(action=...)` 반환 시. reason 은 `signal.reason` 또는 `"entry"`.
  - `"hold"` reason `"no_signal"` — strategy 가 `None` 반환.
  - `"hold"` reason `"action_hold"` — `Signal(action="hold")` 반환.
  - `"exception"` reason 은 exception class name (예: `"RuntimeError"`).
- emit 메서드: `_emit_strategy_evaluated(sid, *, symbol, decision, reason, ts)` — `_emit_strategy_toggled` 패턴 차용. observer 예외 swallow + warn.
- 운영 가시화 목적: live-daemon 의 strategy 가 실제로 호출되는지 검증 (AC0_strategy_dispatch). 4일째 paper 발주 0건 같은 케이스에서 "on_bar 호출 0건" vs "호출됐지만 hold" 분리 진단 가능.
- 측정 게이트: AC5 — 24h 운영 후 events ≥ 1000 (#231).
