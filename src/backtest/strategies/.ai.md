### Live-scanner strategies (#227)

- `_indicators.py` (2026-05-26 v0.6.0) — 진입 필터용 지표 헬퍼. `ema()`, `adx(period=14)`, `choppiness_index(period=14)`, `hurst_exponent(lookback=100)` 4 순수 함수. 외부 의존성 없음 (numpy + pandas). 각 함수는 데이터 부족 시 `None` 반환 (raise 안 함) — 호출자는 "indicator 미사용 → 필터 적용 안 함" 으로 해석해 warmup 단계 차단을 막는다. 단위 테스트 14건 `tests/backtest/test_indicators.py`.
- `_live_scanner_helpers.py` — `LiveScannerMixin` 마커 클래스. `is_live_scanner=True` + `stop_loss_pct` / `take_profit_pct` / `trailing_stop_pct` 클래스 속성. 이 마커 inherit 하면 `AsyncStrategyOrchestrator.run_bar` 가 `market_snapshot["ohlcv_history"]` 의 각 symbol 마다 `on_bar(ctx)` 를 dispatch — per-symbol stateless 평가. 청산은 strategy 가 아니라 `LivePositionRiskManager` (S2 예정) 가 책임. **`_apply_roi_targets` (2026-05-22)** — 레버리지 트레이딩용 ROI 기반 익절/손절. `__init__` 이 `take_profit_roi`/`stop_loss_roi`/`leverage` kwargs 를 받아 호출하면 `take_profit_pct = ROI / leverage` (가격 기준) 로 환산해 덮어쓴다. 예: ROI 12% @ 10x → 가격 1.2%. ROI 인자 미지정 시 no-op (정적 pct 동작 보존). 4개 cand-c live-scanner 전략 공통 적용 — production.yaml 이 ROI/leverage 로 설정. **`_check_entry_filters` (2026-05-26 v0.6.0 A+B+C)** — `on_bar` warmup 직후 호출하는 단일 게이트, 내부적으로 `_check_data_anomaly` (B: last_close/ATR 0 차단) + `_check_trend_filter` (A: ADX ≥ adx_threshold AND close ≥ EMA(slow)) + `_check_regime_filter` (C: Hurst/Choppiness 가 regime_preference 와 일치) 순서로 평가. 각 전략은 `__init__` 에 8 신규 kwargs (anomaly_guard_enabled/trend_filter_enabled/regime_filter_enabled/regime_preference/adx_threshold/ema_slow_period/hurst_lookback/chop_period) 받아 `_apply_filter_kwargs` 호출. 전부 기본 False (anomaly 만 True) — production.yaml 에서 켠다. **`regime_preference` ClassVar 매핑**: breakout=`trend`, bb_lower_bounce=`meanrev`, rsi_oversold=`meanrev`, oversold_with_divergence=`meanrev`. 4일 누적 -312 USDT (5/22~25 BTC false-breakout/INJ over-trade) 대응. 단위 테스트 17건 `tests/backtest/test_live_scanner_filters.py`.

## 리스크 연동 (필수 — #70 이후)

모든 전략은 일수익률 시계열을 **포트폴리오 리스크 모듈로 공급** 해야 한다. 이 규칙은 선택이 아니라 **구현 계약** 이다.

### 어떻게
```python
from portfolio import StrategyOrchestrator
orch.register_strategy_returns("<strategy_id>", daily_return_series)
orch.refresh_portfolio_risk()
```

- `daily_return_series: pd.Series` — index=날짜, 값=그날 실현 수익률
- `strategy_id` = 전략 파일명 snake_case (예: `momo_btc_v2`)

### 왜
- `src/risk/portfolio.py` 의 LW shrinkage Σ / Meucci ENB / Historical CVaR 는 T×N 수익률 매트릭스를 입력으로 받음 ([[19-portfolio-risk]] §2.2, §3.1, §4.1).
- 전략이 수익률을 공급하지 않으면 `portfolio_risk is None` → 리스크 평가기가 **항상 ALLOW 경로** → 리스크 관리 무력화.
- 본 프로젝트는 **단일 evaluator 경로** (`risk.evaluate`) 유지 — 전략마다 자기 리스크 룰을 재구현하면 `rule_id` 라벨 공간 오염.

## 메타라벨링 훅 추가됨 (#85)

`momo_btc_v2.py` 는 LightGBM 2차 메타라벨러 옵션을 지원한다 (opt-in, 기본 disabled).

### 시그니처 확장 불변식
- `metalabeler=None` 이 기본값 — 기존 호출 경로의 시그니처·동작 변경 없음, 회귀 없음
- `metalabeler_threshold: float = 0.5` — 임계값도 opt-in

### 동작
```python
strategy = MomoBtcV2(metalabeler=fitted_ml, metalabeler_threshold=0.5)
# metalabeler=None (기본) → bypass, 기존 동작 보존
```

- `p_take >= threshold` → 신호 통과, `Signal.win_probability = p_take`
- `p_take < threshold`  → `Signal(action="hold", reason="metalabeler_reject", win_probability=p_take)`

### 전략 시그니처 확장 규칙
새 opt-in 파라미터를 전략 `__init__` 에 추가할 때 **반드시 기본값을 기존 동작과 동일하게** 유지해야 한다. 기존 테스트를 수정하지 않고 통과해야 회귀 없음이 보장된다.

### 프로덕션 구성
메타라벨러 프로덕션 활성화 시 `configs/orchestrator/production.yaml` 참조. 온/오프 이원화 등록 + A/B 모니터링 지표는 `docs/specs/strategies/momo-btc-v2.md` 의 "프로덕션 구성" 섹션 참조.
