## 인수 형식 (모두 선택사항)

`$ARGUMENTS`는 다음 형식 중 하나거나 생략 가능하다:

| 형식 | 예시 | 동작 |
|------|------|------|
| 생략 | (없음) | 현재 브랜치에서 자동 감지 |
| 이슈번호만 | `155` | 이슈번호로 브랜치/워크폴더 탐색 |
| 6자리 패딩+짧은이름 | `000155-cross-asset-compare` | 직접 경로 구성 |
| 짧은이름만 | `cross-asset-compare` | 브랜치 목록에서 매칭 |

## 실행 순서

### 1. 컨텍스트 수집

`/remind-issue` 와 동일한 방식으로 이슈번호·브랜치·WORKFOLDER 를 확정한다.

병렬로 실행:
1. `gh issue view {이슈번호} --json title,body` — 이슈 본문
2. `{WORKFOLDER}/00_issue.md` 가 있으면 Read
3. `{WORKFOLDER}/01_plan.md` 가 있으면 Read
4. `{WORKFOLDER}/02_implementation.md` 가 있으면 Read
5. `git log --oneline -10 {브랜치명}` (브랜치가 있는 경우)
6. `git diff master...{브랜치명} --stat` (브랜치가 있는 경우)
7. `git status --short`

전부 없으면 (워크폴더·브랜치 미존재) 이슈 본문만으로 설명한다.

### 2. 설명 작성 가이드

**원칙**:
- 고등학생 1명 (코딩 경험 없음, 주식·암호화폐 모름) 을 청자로 가정
- 영어 약어·전문용어 등장 시 즉시 풀어쓰기 (예: "API" → "다른 프로그램이랑 데이터 주고받는 통로")
- 비유 1개 이상 필수 — 일상에서 만질 수 있는 것 (자동차, 게임, 학교 시스템 등)
- 파일 경로·커밋 해시·함수명 등 코드 식별자 노출 금지
- 우리 프로젝트가 뭐 하는 건지 (자동매매 봇) 1줄로 깔리고 시작
- 분량: 화면 1-2 스크롤 이내

**금지 단어 (반드시 풀어쓸 것)**:
- API, SDK, JSON, manifest, schema, backtest, manifest, equity-curve, Sharpe, DSR, regime, HMM, PR, 머지, 리팩터링, 레포지토리, 워크트리, 브랜치, AC, ADR, CVaR, walk-forward, ablation, bootstrap, p5, fold, kill-switch, dry-run, fallback

→ 위 단어가 본문에 등장하면 모두 일상 표현으로 치환한 뒤 출력한다.

**용어 치환 예시**:
- "백테스트" → "옛날 데이터로 미리 돌려보는 모의실험"
- "머지" → "다른 사람 코드와 합치기"
- "Sharpe" → "수익을 위험으로 나눈 점수"
- "regime" → "시장 분위기 (강세장/약세장 같은 거)"
- "kill-switch" → "비상정지 버튼"
- "API 키" → "거래소 비밀번호 같은 거"

```

  "{이슈 제목을 일상어로 의역한 한 줄}"
────────────────────────────────────────

🎯 뭘 풀려고 했냐면

{2-3문단. 비유 1개 포함. 다음 흐름 권장:}
{- 우리 프로젝트는 ___ 하는 봇이야}
{- 근데 거기서 ___ 가 안 됐어 / 잘못됐어}
{- 그래서 ___ 가 필요했어}

🛠️ 무슨 작업을 했냐면

{2-3문단. 결과 중심. 다음 흐름 권장:}
{- ___ 부분을 ___ 처럼 바꿨어}
{- 그래서 이제 ___ 가 가능해졌어 (또는 가능해질 예정이야)}

📊 지금 어디까지 왔냐면

  ✅ 끝난 거: {완료된 항목 1-2개를 일상어로}
  🔧 남은 거: {미완료 항목 1-2개를 일상어로}
  📍 다음 할 일: {다음 단계 1줄}

(작업 시작 전이면 "🛠️ 무슨 작업을 했냐면" 섹션을 다음으로 대체)

🛠️ 어떻게 풀 계획이냐면
{01_plan.md 또는 이슈 본문 기반으로 계획을 일상어로 2문단}
```

### 4. 후처리

- 출력 후 추가 작업 없음 (파일 수정·커밋 금지)
- 사용자가 더 자세한 설명 요청하면 같은 형식으로 더 풀어서 답한다

## 예시

**입력**: `/hs 155`

**기대 출력 톤** (참고용 — 실제 내용은 이슈마다 다름):

```
🎓 이슈 #155 — 고등학생용 요약

  "성적표 두 개를 한 페이지에 비교해서 보고 싶은데, 한쪽 성적표가 자꾸 빈 칸으로 나와요"

🎯 뭘 풀려고 했냐면

우리 프로젝트는 주식이랑 암호화폐를 자동으로 사고파는 봇이야. 어떤 매매 전략이 더 잘 먹히는지 알려면, 옛날 데이터로 미리 돌려본 결과를 표로 비교해야 해.

근데 한국 주식 쪽 결과가 자꾸 "데이터 없음"으로 나왔어. 결과는 분명히 만들어졌는데, 비교하는 프로그램이 그걸 못 찾아 읽어오는 거야.

🎯 비유하자면, A반·B반 시험 점수를 한 페이지에 비교하려고 하는데, B반 선생님이 점수표를 책상 서랍에 넣어두고 "복도 게시판"에 붙여놓지 않아서 비교 담당자가 못 보는 상황이야.

🛠️ 어떻게 풀 계획이냐면

옵션 두 가지가 있어:
1. B반 선생님이 게시판에 붙이게 만든다 (= 결과를 정해진 폴더에 파일로 저장)
2. 비교 담당자가 책상 서랍을 직접 열어보게 만든다 (= 결과를 직접 읽어들이게)

1번이 더 깔끔해. 다른 반(BTC)은 이미 그렇게 하고 있거든.

📊 지금 어디까지 왔냐면

  📍 다음 할 일: 의존하는 다른 작업이 먼저 합쳐지길 기다리는 중
```

## 주의사항

- 본 커맨드는 **읽기 전용**: 어떤 파일도 수정하지 않는다.
- AC 자동 체크나 `00_issue.md` 갱신은 하지 않는다 (그건 `/remind-issue` 의 역할).
- 영문 약어가 도저히 풀어쓸 수 없으면 (제품 고유명사 등) "○○" 표기 후 처음 한 번만 옆에 짧게 풀이를 단다.
