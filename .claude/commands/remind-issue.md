## 인수 형식 (모두 선택사항)

`$ARGUMENTS`는 다음 형식 중 하나거나 생략 가능하다:

| 형식 | 예시 | 동작 |
|------|------|------|
| 생략 | (없음) | 현재 브랜치에서 자동 감지 |
| 이슈번호만 | `15` | 이슈번호로 브랜치 탐색 |
| 6자리 패딩+짧은이름 | `000015-my-feature` | 직접 경로 구성 |
| 짧은이름만 | `my-feature` | 브랜치 목록에서 매칭 |

## 실행 순서

### 1. 브랜치·이슈번호 확정

**인수 없는 경우:**
```
git branch --show-current
```
결과 예: `feat/000015-pdf-upload` → PADDED=`000015`, 짧은이름=`pdf-upload`, 이슈번호=`15`

**이슈번호(정수)만 있는 경우:**
```
git branch | grep {zero-padded 번호}
```

**`000015-my-feature` 또는 짧은이름 형식:**
```
git branch | grep {패턴}
```

매칭 없으면 오류 안내 후 중단.

- WORKFOLDER = `docs/work/active/{PADDED}-{짧은이름}`

### 1.5. 터미널/tmux 윈도우 컨텍스트 설정

Step 1에서 PADDED · 짧은이름 추출 직후 실행한다:

```bash
if [ -n "$TMUX" ]; then
  tmux rename-window -t "$TMUX_PANE" "{이슈번호}-{짧은이름}"
else
  printf '\033]0;#%s %s\007' "{이슈번호}" "{짧은이름}"
fi
```

- tmux 내: `$TMUX_PANE`으로 현재 pane의 윈도우를 명시 타겟팅해 이름 변경
- tmux 밖: 터미널 타이틀을 `#{이슈번호} {짧은이름}` 으로 변경
- 실패 시 오류만 출력하고 계속 진행한다.

---

### 2. AC 목록 추출

`{WORKFOLDER}/00_issue.md`가 존재하면 읽어서 `## 완료 기준` 또는 `## AC` 섹션의 항목을 추출한다.

파일이 없으면:
```
gh issue view {이슈번호} --json body --jq '.body'
```
로 이슈 body에서 AC 섹션을 추출한다.

AC가 없으면 AC 섹션 없이 변경 파일·다음 작업만 출력한다.

### 2.5. 구현 계획 검증 및 강제 작성

`{WORKFOLDER}/01_plan.md`를 읽는다 (파일 없으면 경고만 표시하고 계속 진행).

`## 구현 계획` 섹션을 검증한다:

1. **섹션 없음 / 비어있음 / `(작성 예정)`** → `/plan` 커맨드를 즉시 실행해 채운다.
2. **간략한 플랜** (아래 최소 요건 미충족) → 경고 출력 후 `/plan` 커맨드를 실행해 구체화한다.

**구체적 플랜 최소 요건** (하나라도 미충족 시 "간략한 플랜"으로 판정):
- AC 각 항목별 구현 단계가 명시되어 있을 것
- 변경/생성 대상 파일·디렉토리가 명시되어 있을 것
- 단계별 실행 순서가 있을 것
- Guardrails (Must Have / Must NOT Have) 또는 주의사항이 있을 것

⚠️ **플랜이 최소 요건을 충족하지 않으면 구현을 시작하지 않는다. 반드시 플랜을 먼저 보강한 뒤 다음 단계로 진행한다.**

`/plan` 완료 후 계속 진행한다.

### 3. AC 자동 체크

Step 2에서 추출한 미완료(`[ ]`) AC 항목 각각에 대해 완료 여부를 판단하고, 완료된 항목은 `00_issue.md`에서 `- [ ]` → `- [x]` 로 업데이트한다.

**판단 방법 — 아래 증거를 조합해 판단한다:**
1. **파일 존재 여부** — AC 항목이 특정 파일 생성을 요구하면 `ls` 또는 Read로 확인
2. **git log** — 커밋 메시지에서 해당 기능 완료 증거 확인
3. **git status** — staged/unstaged 변경사항으로 작업 완료 추론
4. **실제 파일 내용** — 항목이 요구하는 내용이 파일에 존재하는지 Read로 확인

**규칙:**
- 증거가 명확할 때만 `[x]` 로 변경한다. 불확실하면 `[ ]` 유지.
- 새로 체크된 항목이 있으면 `00_issue.md`를 Edit로 업데이트한다.
- 체크된 항목 목록을 다음 형식으로 출력한다:
  ```
  ✅ AC 자동 체크: {체크된 항목 요약} → [x] 처리
  (변경 없음 — 미완료 항목 유지)
  ```

> **주의**: 판단은 추정이며, 동작 테스트나 실행 결과가 아닌 코드 구조 기준입니다. 일부 AC는 직접 확인이 필요할 수 있습니다.

### 4. 현재 단계 판단

```bash
git status --short
```

1. 미커밋 변경사항 있으면 → **"구현 중"**
2. 미커밋 없고 AC 미완료 항목 있으면 → **"구현 대기"**
3. AC 항목 전체 `[x]` 이면 → **"완료 — /finish-issue 실행 필요"**

`01_plan.md`의 `## 구현 계획`에 스텝이 있으면, 현재 단계 옆에 첫 번째 미완료 스텝 이름도 함께 표시한다.

### 5. 현황 출력

다음 형식으로 출력한다:

```
────────────────────────────────────────
이슈 #{번호} — {제목}
브랜치: {브랜치명}
────────────────────────────────────────
AC 진행 상황 ({완료수}/{전체수} 완료):
  [x] 완료된 항목
  [ ] 미완료 항목

📍 현재 단계: {구현 중 | 구현 대기 | 완료}
  ({01_plan.md 첫 번째 미완료 스텝, 없으면 생략})

🔀 최근 커밋 (git log --oneline -3):
  {커밋 목록, 없으면 "없음"}

📁 미커밋 변경사항 (git status --short):
  {변경 목록, 없으면 "없음"}
────────────────────────────────────────
{AC 전체 완료 시에만 출력:}
🎉 모든 AC 항목이 완료되었습니다!
👉 PR을 올릴 준비가 됐다면: /fi 또는 /finish-issue
────────────────────────────────────────
```

### 6. 작업 내역 기록

`{WORKFOLDER}/00_issue.md`가 존재하면 `## 작업 내역` 섹션에 현황 스냅샷을 추가한다.

오늘 날짜 항목이 이미 있으면 **해당 항목을 업데이트**한다 (skip 아님).

추가 형식:
```
### {오늘 날짜}

**현황**: {완료된 AC 수}/{전체 AC 수} 완료
**완료된 항목**:
- {완료 AC 목록}
**미완료 항목**:
- {미완료 AC 목록}
**변경 파일**: {변경된 파일 수}개
```
